# `payment` convex feature

Per-provider Convex tables. Each payment provider (Midtrans, Doku, Stripe…)
owns its own namespaced table set so two providers can coexist in a single
Convex deployment without the legacy `paymentOrders` collision.

## Tables (post 2026-05-14 namespace split)

- `doku_orders` / `doku_webhook_events` — provided by `doku-payment` slice.
- `midtrans_orders` / `midtrans_webhook_events` — provided by `midtrans-payment` slice.

The legacy shared shape (`paymentOrders` / `paymentWebhookEvents`) is
retained only as a migration source — see below.

## Migration from the pre-namespace shape

Track G (Wave N+1, 2026-05-14) resolved the documented collision between
`doku-payment` and `midtrans-payment`. Two one-shot rename mutations were
generated by the Phase-E migration planner and live at:

- `convex/migrations/M-doku-namespace-2026-05.ts`
  - copies `paymentOrders` → `doku_orders`
  - copies `paymentWebhookEvents` → `doku_webhook_events`
- `convex/migrations/M-midtrans-namespace-2026-05.ts`
  - copies `paymentOrders` → `midtrans_orders`
  - copies `paymentWebhookEvents` → `midtrans_webhook_events`

Each provider's rename is independent — the legacy shared rows split into
the two new table sets based on their `provider` discriminator. Operator
runbook:

1. Back up the legacy `paymentOrders` / `paymentWebhookEvents` tables.
2. Run each rename `internalMutation` from the Convex dashboard. The
   migrations are reversible (no `drop` step) until you explicitly delete
   the legacy tables.
3. Verify row counts for each provider match the prior shared-table count.
4. Drop the legacy tables in a follow-up irreversible migration once
   verified.

## Add a new provider (e.g., Stripe)

1. Author a `slice.contract.ts` with `requires.convex.prefix = "stripe_"`
   and `provides.tables = ["stripe_orders", "stripe_webhook_events"]`.
2. Create `convex/features/payment/actions/stripe.ts` mirroring the
   existing providers' shape (export `createTransaction`).
3. Register the namespaced tables in `convex/features/payment/_schema.ts`
   under a `stripeTables` export, and spread into `convex/_schema.ts`.
4. Add a frontend wrapper in the slice's components directory.

Per-provider tables keep blast radius low: a Stripe schema bug can't
corrupt Doku rows, and the compose-solver green-lights any pair of
properly-namespaced providers without further migration work.
